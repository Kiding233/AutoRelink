## 八、AutoRelink 插件开发实录

> 🎯 **本章目标**：记录 AutoRelink 插件的完整开发过程，包括算法设计、问题发现、性能优化等实战经验

### 8.1 算法设计

#### 八方向区域划分算法

基于节点的四条边延长线将平面分为8个区域：

```
        topLeft │     top     │ topRight
        ────────┼────────────┼─────────
         left   │   节点本身   │  right
        ────────┼────────────┼─────────
      bottomLeft │   bottom   │ bottomRight
```

判断优先级：先四角 → 后四边 → 最后中心（按 dx/dy 判断）

#### 历史记录机制

- 每个（源节点, 目标节点）对维护一个长度为2的历史队列
- 仅当区域变化时才更新历史
- 历史格式: `[prevRegion, currentRegion]`，初始为 `[null, firstRegion]`

#### 方向解析逻辑

| 历史组合 | 结果方向 |
|---------|---------|
| [null, cardinal] | cardinal（首次）|
| [null, diagonal] | 转为最近的 cardinal |
| [cardinal, diagonal] | 取 cardinal（稳定）|
| [diagonal, cardinal] | 取 cardinal（切换）|
| [cardinal1, cardinal2] | 取 cardinal2（最新）|

#### 连线端点调整流程

```
tick() 循环（60fps）
  ↓
获取选中节点 → 获取其矩形坐标
  ↓
遍历关联节点 → 计算相对区域 → 更新历史
  ↓
如果历史变化 → calcRates() 计算端点位置
  ↓
设置 sourceRectangleRate / targetRectangleRate
  ↓
updateReferences() 刷新显示
```

#### 🔴 重要发现：节点瞬移是软件本体问题

**症状**: 拖拽选中节点时，节点突然跳到右下角或其他异常位置

**结论（经实测验证）**: **这不是插件导致的问题，而是 Project Graph 软件本体的已知行为。**

##### 原因分析

Project Graph 在处理某些端点值或布局刷新时可能出现位置计算异常：
- 可能与 `updateReferences()` 触发的内部布局重算有关
- 可能与软件的自动布局算法冲突
- 在特定操作序列下触发（非100%复现）

##### 插件的防护措施（仍有价值）

虽然瞬移是软件本体问题，但插件仍应避免使用极端值：

```javascript
// ✅ 推荐：始终使用4个主方向的边缘中心值
{ x: 0.99, y: 0.5 }   // right
{ x: 0.01, y: 0.5 }   // left  
{ x: 0.5, y: 0.01 }   // top
{ x: 0.5, y: 0.99 }   // bottom

// ⚠️ 避免使用角落值（可能增加软件本体异常的概率）
{ x: 0.99, y: 0.99 }   // bottomRight ← 不推荐
```

### 8.3 性能优化历程

#### 大图性能问题与解决方案

**症状**: 图中有几十上百个节点时，AutoRelink 完全不响应或 `moved=false`

##### 原因分析

1. **tick() 执行时间过长**
   - 大量节点 → 大量边 → `getRelated()` 遍历耗时
   - 每个关联节点都要 `getRect()` (7次 await)
   - 单次 tick 可能超过 16ms (60fps) 甚至数百毫秒

2. **移动检测失效**
   - 三阶段初始化需要连续两帧完成
   - 大图中两帧间隔可能很长（因为 tick 执行慢）
   - 5px 移动阈值在采样间隔大时难以触发

3. **`moved=false` 导致 adjust() 不执行**

##### 解决方案：强制调整机制

```javascript
const FORCE_ADJUST_INTERVAL = 500;  // 每500ms强制调整一次
let lastForceAdjustTime = 0;

// 在 tick() 中：
const forceAdjust = !moved && (Date.now() - lastForceAdjustTime > FORCE_ADJUST_INTERVAL);
if (forceAdjust) lastForceAdjustTime = Date.now();

// 触发条件改为：
if (moved || forceAdjust) {
  await adjust(...);
}
```

##### 其他优化措施

| 措施 | 效果 |
|------|------|
| 降低 MOVE_THRESHOLD (5→3) | 更灵敏地检测移动 |
| init→false 时立即 moved=true | 第2帧就允许调整 |
| allEdges 缓存（只获取1次） | 减少 ~50% 边遍历 |
| relRects 预计算 Map | 避免 adjust() 中重复 getRect() |

#### 🔬 Entity 边获取方法实测结果

##### 测试背景

大图性能问题的根源是 `getRelated()` 每次调用都遍历全部边（O(E)复杂度）。
理论上 Entity/StageManager 应该有直接的边获取方法（如 `getOutgoingEdges()`），可以避免遍历。

##### 源码分析

通过阅读 `project-graph-3.0.0` 源码发现：

**ConnectableEntity 类** (`app/src/core/stage/stageObject/abstract/ConnectableEntity.tsx`):
```typescript
export abstract class ConnectableEntity extends Entity {
  abstract geometryCenter: Vector;
  public unknown = false;
}
// ⚠️ 非常简单！没有任何边相关属性或方法
```

**GraphMethods 类** (`app/src/core/stage/stageManager/basicMethods/GraphMethods.tsx`):
```typescript
public getOutgoingEdges(node: ConnectableEntity): Edge[] {
  const result: Edge[] = [];
  for (const edge of this.project.stageManager.getEdges()) {
    if (edge.source === node) {   // ← 用 === 比较对象引用！
      result.push(edge);
    }
  }
  return result;
}
// ⚠️ 内部还是遍历全部边！且用 === 比较，对 Comlink Proxy 可能失效
```

##### 实测结果

在 Web Worker 环境中测试了以下方法：

| 方法/属性 | 类型 | 结果 | 原因 |
|-----------|------|------|------|
| `ent.getOutgoingEdges()` | ❌ undefined | **方法不存在于 Proxy 上** |
| `sm.getOutgoingEdges(ent)` | ❌ undefined | **StageManager 未暴露此方法** |
| `ent.getIncomingEdges()` | ❌ undefined | 同上 |
| `ent.edges()` | ⚠️ function 但返回 undefined | Comlink 函数上下文问题 (`.apply` 失败) |

##### 结论：**没有可用的直接边获取 API**

Entity 对象上没有暴露任何边获取方法给插件使用。
源码中的 `getOutgoingEdges()` 在 GraphMethods 类上是 `public` 的，
但 StageManager 通过 Comlink 暴露给插件的 API 子集不包含这些方法。

#### 🚀 最终方案：索引缓存 + O(1) 查找

既然无法避免遍历全部边建索引，那就**只建一次并缓存**：

##### 核心数据结构

```javascript
// 边索引: Map<UUID, { out: Edge[], in_: Edge[] }>
edgeIndex = {
  "uuid_A": { out: [edge1, edge2], in_: [edge3] },
  "uuid_B": { out: [edge3], in_: [edge1, edge4] },
  ...
}
```

##### 三层缓存策略

```
tick() 调用 getEdgeIndex(allEdges)
    │
    ├─ 条件1: cachedEdgeIndex === null（首次）
    │   → 建立索引 → 缓存 → 返回
    │
    ├─ 条件2: allEdges.length !== cachedEdgeCount（边数变化）
    │   → 重建索引 → 更新缓存 → 返回
    │   触发时机: 新增/删除连线时
    │
    ├─ 条件3: now - lastIndexBuildTime > INDEX_CACHE_TTL（缓存过期）
    │   → 强制刷新 → 防止遗漏
    │   TTL = 2000ms（2秒）
    │
    └─ 其他情况 → 直接返回缓存 ✅ 零开销
```

##### 完整代码

```javascript
let cachedEdgeIndex = null;
let cachedEdgeCount = -1;
let lastIndexBuildTime = 0;
const INDEX_CACHE_TTL = 2000;  // 2秒缓存

async function getEdgeIndex(allEdges) {
  const now = Date.now();
  if (cachedEdgeIndex === null 
      || allEdges.length !== cachedEdgeCount 
      || now - lastIndexBuildTime > INDEX_CACHE_TTL) {
    
    // 需要重建
    cachedEdgeIndex = await buildEdgeIndex(allEdges);
    cachedEdgeCount = allEdges.length;
    lastIndexBuildTime = now;
  }
  return cachedEdgeIndex;
}

async function buildEdgeIndex(allEdges) {
  const idx = new Map();
  for (const e of allEdges) {
    try {
      const s = await e.source.uuid;
      const t = await e.target.uuid;
      if (!idx.has(s)) idx.set(s, { out: [], in_: [] });
      if (!idx.has(t)) idx.set(t, { out: [], in_: [] });
      idx.get(s).out.push(e);
      idx.get(t).in_.push(e);
    } catch(x) {}
  }
  return idx;
}

function getRelatedFromIndex(idx, uid) {
  return idx.get(uid) || { out: [], in_: [] };  // O(1)
}
```

##### 性能对比（1000条边的图）

| 场景 | 之前（每帧遍历） | 现在（索引缓存） | 提升 |
|------|-----------------|---------------|------|
| **每帧异步操作数** | **2000次** (1000×2) | **~16次** (仅首帧) | **125x ↓** |
| **1秒内总异步数** | **120,000次** | **~500-2000次** | **60x ↓** |
| **getRelated() 复杂度** | **O(E)** 遍历 | **O(1)** Map.get | **∞** ↑ |
| **getRelated() 是否 async** | 是（有 await） | **否（同步）** | 去掉 await |
| **选中后卡顿** | ⚠️ 大图可能卡顿 | ✅ 流畅 | - |
| **内存占用** | 无额外 | ~50KB Map | 可忽略 |

##### tick() 中的调用链

```
tick()
  │
  ├─ sm.getEdges()                    ← 1次异步（必须）
  │
  ├─ getEdgeIndex(allEdges)           ← 带缓存的索引获取
  │   ├─ [首次/边变/过期] → buildEdgeIndex() → 遍历全部边建Map
  │   └─ [其他125帧] → 直接返回缓存       ← 零开销 ✅
  │
  └─ for each selected node:
       ├─ getRect(ent)                 ← 7次异步
       ├─ getRelatedFromIndex(idx, uid) ← 0次异步! O(1) ✅
       ├─ for each related node:
       │    └─ getRect(r)               ← 7次异步
       └─ if moved||forceAdjust:
            └─ adjust(...)              ← 用预计算的 relRects
```

### 8.4 🚀 v2.0 极致性能优化（十项优化措施）

> 🎯 **本章目标**：解决优化后仍存在的1-2秒延迟，通过深度分析瓶颈实施10项优化，将响应时间降低到100-300ms级别

#### 8.4.1 性能瓶颈深度诊断

##### 核心问题：`getRect()` 函数是最大瓶颈

```javascript
async function getRect(e){
  const r = await e.collisionBox.getRectangle();  // ① 1次IPC
  const l = await r.left,                          // ② 1次IPC
      right = await r.right,                       // ③ 1次IPC
      t = await r.top,                             // ④ 1次IPC
      b = await r.bottom,                          // ⑤ 1次IPC
      cx = (await r.center.x),                     // ⑥ 1次IPC（嵌套属性！）
      cy = (await r.center.y);                     // ⑦ 1次IPC（嵌套属性！）
  return {left:l, right:right, top:t, bottom:b, center:{x:cx, y:cy}};
}
```

**每次调用 = 7次跨线程Comlink IPC通信** ⚠️

##### tick() 完整异步调用链分析（典型场景：1个选中节点 + 20条关联边）

| 操作位置 | 操作内容 | IPC次数 | 预估耗时 |
|---------|---------|--------|---------|
| `sm.getSelectedEntities()` | 获取选中节点 | 1次 | ~15ms |
| `sm.getEdges()` | 获取所有边 | 1次 | ~15ms |
| `getRect(ent)` | 选中节点矩形 | **7次** | ~105ms |
| **获取关联实体** (串行) | 20条边的target/source | **40次** | **~600ms** |
| `getRect(r)` | 20个关联节点矩形 | **140次** | **~2100ms** |
| **总计** | - | **~190次** | **~2850ms (2.85秒)** |

**这就是1-2秒延迟的根本原因！** 🎯

##### ⭐ 优化6：矩形缓存策略（**效果最大！预期减少70-90%延迟**）

**核心思想**：节点位置不会每帧都变，缓存矩形并在移动时才刷新

```javascript
// 全局矩形缓存
let rectCache = new Map();  // uuid → { rect, timestamp }
const RECT_CACHE_TTL = 150;  // 缓存有效期150ms

async function getCachedRect(uuid, entity, forceRefresh=false) {
  const cached = rectCache.get(uuid);
  const now = Date.now();
  
  // 命中缓存：0次IPC ✅
  if (!forceRefresh && cached && (now - cached.timestamp) < RECT_CACHE_TTL) {
    return cached.rect;
  }
  
  // 未命中：7次IPC
  const rect = await getRect(entity);
  rectCache.set(uuid, { rect, timestamp: now });
  return rect;
}
```

**使用方式**：
- 选中节点：`getCachedRect(uid, ent, true)` → 强制刷新（因为正在拖拽）
- 关联节点：`getCachedRect(ruid, r)` → 使用缓存（通常不移动）

**效果预估**：
- 静态关联节点：100%命中缓存 → **0次额外IPC**
- 移动中的选中节点：150ms内只获取1次 → **减少90%重复调用**
- **总体延迟降低：70-90%**

##### ⭐ 优化7：Promise.all 并行化（**提速5-20倍**）

**核心思想**：独立的异步操作应该并行执行，而非串行等待

**之前（串行）**：
```javascript
// ❌ 20条边需要 20×30ms = 600ms
for(const e of out){
  const t = await e.target;     // 等待完成
  const tu = await t.uuid;      // 再等待
  relUUIDs.push(tu);
}
```

**之后（并行）**：
```javascript
// ✅ 20条边只需 ~30ms
const outTasks = out.map(async(ed) => {
  try {
    const tu = ed.tgtUUID;        // 使用预存的UUID（优化8）
    const t = ed.tgtEntity;        // 使用预存的实体引用（优化8）
    return { uuid: tu, entity: t };
  } catch(x) { return null; }
});
const outResults = (await Promise.all(outTasks)).filter(Boolean);

// 同理处理入边
const inTasks = ia.map(async(ed) => { ... });
const inResults = (await Promise.all(inTasks)).filter(Boolean);
```

**效果预估**：
- 20条边：600ms → **30ms**（**提速20倍**）
- 50条边：1500ms → **50ms**（**提速30倍**）

##### ⭐ 优化8：预存实体引用到索引（**每条边减少2次IPC**）

**核心思想**：在`buildEdgeIndex()`时就保存entity引用，避免tick()中重复获取

```javascript
async function buildEdgeIndex(allEdges){
  const idx = new Map();
  for(const e of allEdges){
    try{
      const s = await e.source.uuid, t = await e.target.uuid;
      if(!idx.has(s)) idx.set(s, {out: [], in_: []});
      if(!idx.has(t)) idx.set(t, {out: [], in_: []});
      
      idx.get(s).out.push({
        e,
        tgtUUID: t,
        tgtEntity: e.target   // 🔑 新增：预存目标实体引用
      });  
      idx.get(t).in_.push({
        e,
        srcUUID: s,
        srcEntity: e.source   // 🔑 新增：预存源实体引用
      });  
    }catch(x){}
  }
  return idx;
}
```

**配合优化7使用**：
```javascript
// 不再需要 await e.target 和 await e.source
const outTasks = out.map(async(ed) => {
  const tu = ed.tgtUUID;     // 直接使用预存值
  const t = ed.tgtEntity;    // 直接使用预存引用
  return { uuid: tu, entity: t };
});
```

**效果预估**：
- 每条边减少2次IPC（`e.target` 和 `e.source`）
- 20条边 = **减少40次IPC ≈ 600ms**

##### ⭐ 优化9：智能跳过未变化的边（**避免80%无效计算**）

**核心思想**：相对区域没变的边不需要重新计算端点

```javascript
let needAdjustList = [];  // 只收集真正需要调整的边

for(const ruid of relUUIDs){
  const rr = await getCachedRect(ruid, r);  // 使用缓存
  const region = getRegionByEdges(rect.center.x, rect.center.y, rr);
  const srKey = `${uid}-${ruid}`;
  const lastReg = lastRegionMap.get(srKey);
  
  if(lastReg === region){
    continue;  // 🔑 区域没变化，跳过此边
  }
  
  lastRegionMap.set(srKey, region);
  if(moved && isCardinal(region) && updHist(uid, ruid, region)){
    changed = true;
    needAdjustList.push({ruid, rr});  // 🔑 收集需调整的边
  }
}

// 只对变化的边执行adjust()
if((moved || forceAdjust) && needAdjustList.length > 0){
  const c = await adjust(..., new Map(needAdjustList.map(x => [x.ruid, x.rr])));
}
```

**效果预估**：
- 稳定拖拽时：80%的边区域不变 → **跳过80%计算**
- 只有穿越边界时才触发完整流程

#### 8.4.4 tick() 新版调用链（v2.0优化后）

```
tick()
  │
  ├─ 复用缓存的 project 和 sm          ← 0次IPC (优化1)
  │
  ├─ sm.getSelectedEntities()          ← 1次IPC
  ├─ sm.getEdges()                     ← 1次IPC
  │
  ├─ getEdgeIndex(allEdges)            ← 带缓存的索引获取 (优化2-5)
  │   └─ [命中缓存] → 直接返回           ← 零开销 ✅
  │
  └─ for each selected node:
       │
       ├─ getCachedRect(uid, ent, true)  ← 7次IPC (选中节点强制刷新)
       │   └─ [150ms内重复] → 返回缓存     ← 0次IPC (优化6)
       │
       ├─ getRelatedFromIndex(idx, uid)  ← 0次IPC! O(1) ✅
       │
       ├─ Promise.all 并行获取关联实体    ← ~30ms (优化7+8)
       │   └─ 使用预存的 UUID + 实体引用   ← 0次额外IPC (优化8)
       │
       ├─ for each related node:
       │    ├─ getCachedRect(ruid, r)     ← 0-7次IPC (优化6)
       │    │   └─ [命中缓存] → 0次IPC     ← 大部分情况 ✅
       │    │
       │    └─ [区域变化?]                ← 智能跳过 (优化9)
       │        ├─ [相同] → continue      ← 跳过80%边 ✅
       │        └─ [不同] → 收集到needAdjustList
       │
       └─ adjust(needAdjustList)         ← 只处理变化的边 (优化9)
            └─ updateReferences()         ← 1次IPC
```

**总IPC次数对比**：

| 版本 | 典型场景(20条边) | IPC次数 | 耗时 |
|------|-----------------|--------|------|
| **v1.0** | 每帧全部遍历 | **~190次** | **~2850ms** |
| **v2.0** | 优化后 | **~30-50次** | **~150-300ms** |
| **提升** | - | **4-6x ↓** | **10-19x ↑** |

#### 8.5.2 工作流程

##### 阶段一：开发阶段（日常）

✅ **所有修改只在开发版本进行**
- 工作目录：`AutoRelink/extension.js`
- 包含完整的调试日志（`DEBUG_MODE = true` 或根据需要开启）

- 确认功能正常
- 验证性能改善
- 检查边界情况

⚠️ **用户必须明确说以下任一命令**：
- "同步到发布版"
- "打包发布版"
- "发布新版本"

✅ **执行的操作**：
1. 将 `AutoRelink/extension.js` 复制到 `AutoRelink_1.0.0/extension.js`
2. 在发布版中：
   - 设置 `DEBUG_MODE = false`（关闭详细日志）
   - 移除或精简调试输出
   - 更新版本号注释
3. 执行打包脚本生成 `.prg` 文件

#### 8.5.3 版本管理建议

| 版本类型 | 文件位置 | 日志级别 | 用途 |
|---------|---------|---------|------|
| **开发版** | `AutoRelink/` | 详细日志(DEBUG_MODE可调) | 日常开发、调试 |
| **发布版** | `AutoRelink_1.0.0/` | 无日志(DEBUG_MODE=false) | 用户使用、分发 |

#### 8.5.4 为什么采用这个流程？

**优势**：
1. **避免意外覆盖**：不会在开发过程中误改发布版
2. **保持发布版稳定**：只有经过充分测试的代码才会进入发布版
3. **灵活的调试控制**：开发版可以随时开关日志
4. **清晰的版本追溯**：明确的开发→测试→发布流程

**注意事项**：
- 发布版的代码应该是开发版的子集（只去掉日志相关内容）
- 核心逻辑完全一致，确保行为相同
- 打包前务必再次测试发布版
